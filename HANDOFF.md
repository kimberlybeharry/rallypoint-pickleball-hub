## 4. Application Architecture

### Data Flow

```
data.ts (static mock data)
    |
    v
cart-context.tsx (React Context + useReducer)
    |                          |
    v                          v
localStorage                 UI components
(hydrate on mount)           (read items/total/count)
```

### State Management

The cart uses `useReducer` with 6 action types:

| Action | Description |
|---|---|
| `ADD_PRODUCT` | Adds product to cart or increments quantity if already present |
| `ADD_BOOKING` | Adds court booking (deduplicated by `booking-{courtId}-{date}-{time}`) |
| `REMOVE_ITEM` | Removes by item ID |
| `UPDATE_QTY` | Adjusts product quantity (removes if qty < 1) |
| `CLEAR` | Empties cart (called after payment) |
| `HYDRATE` | Restores cart from localStorage on mount |

Cart persists to `localStorage` key `"rp-cart"` on every state change.

### Cart Item Types

```typescript
// Product in cart
interface CartProduct {
  type: "product";
  id: string;           // "prod-{productId}-{timestamp}"
  product: Product;     // Full product object from data.ts
  quantity: number;
}

// Court booking in cart
interface CartBooking {
  type: "booking";
  id: string;           // "booking-{courtId}-{date}-{time}"
  courtId: string;
  courtName: string;
  date: string;         // "YYYY-MM-DD"
  time: string;         // "HH:MM"
  timeLabel: string;    // "7:00 AM"
  price: number;        // Always 15.00
}
```

### Page Responsibilities

| Page | Route | Client/Server | Key Features |
|---|---|---|---|
| Landing | `/` | Server component | Hero banner, 3 feature cards, how-it-works steps, CTA |
| Store | `/store` | Client (`"use client"`) | Category filter pills, text search, add-to-cart with 1.5s "Added" flash |
| Booking | `/booking` | Client (`"use client"`) | Week date picker with prev/next, 3-court x 12-slot grid, pre-booked/in-cart states |
| Checkout | `/checkout` | Client (`"use client"`) | Cart items with qty controls, order summary sidebar, mock Stripe card, 1.8s processing animation, confirmation screen with points earned |
| Dashboard | `/dashboard` | Server component | Points balance (450), Silver-to-Gold progress bar (83%), recent activity list, recommendations, active challenge tracker |

## 12. Tailwind v4 Notes (Important)

This project uses **Tailwind CSS v4**, which works differently from v3:

- **No `tailwind.config.js`** -- Configuration is done via CSS (`@theme` directives in `globals.css`)
- **PostCSS plugin is `@tailwindcss/postcss`** (not the old `tailwindcss` plugin)
- **Import syntax**: `@import "tailwindcss"` in CSS (not `@tailwind base/components/utilities`)
- **Theme customization**: Uses `@theme inline { }` blocks in CSS
- All utility classes work the same as v3, but the config mechanism is completely different

If you encounter build errors related to Tailwind, check that:
1. `postcss.config.mjs` uses `"@tailwindcss/postcss": {}`
2. `globals.css` starts with `@import "tailwindcss";`
3. Both `tailwindcss` and `@tailwindcss/postcss` are in devDependencies

### 15.1 AI Recommendation Engine

From Kimberly's RMET A2 report and A1 proposal: a **hybrid recommender** combining two signal types:

- **Content-based signals**: product attributes (category, price band, skill level, text features), slot attributes (time of day, day of week, court)
- **Collaborative signals**: other users with similar behaviour (click, save, cart, purchase, booking search, prior time-slot choices)
- **Output**: ranked list of both products and court booking offers personalised per user
- **Explainability**: lightweight "because you booked Court 1 on Saturday mornings" style labels on recommendations
- **Privacy**: opt-in personalisation only; transparent data-use notices; minimal data collection

In the prototype, recommendations are static (`RECOMMENDATIONS` array in `data.ts`). The NETLCO agent should plan to replace this with a real recommendation API route backed by a scoring function.

### 15.2 Gamification Rules Engine

From the A1 proposal and A2 report: an **event-driven ledger** with the following defined rules:

**Earning rules (points awarded on event):**

| Event | Points Awarded |
|---|---|
| Account creation (welcome bonus) | 1,000 pts |
| Court booking completed | 50 pts |
| Equipment purchase | 1 pt per $1 USD spent |
| Subscription purchase | 2 pts per $1 USD spent (double) |
| Referral (referred user completes first purchase or booking) | 100 pts |

**Tier thresholds:**

| Tier | Points Required | Benefits |
|---|---|---|
| Silver | 200 | Unlock Silver rewards |
| Gold | 500 | Unlock Gold rewards |

**Challenges (time-bounded missions):**
- Example active challenge: "Complete 3 court bookings this month -- reward: 200 bonus points"
- Progress tracked automatically on each booking event
- Automated reward issued on completion
- Reversal logic: points reversed if a booking is cancelled within the refund window

**Point redemption:**
- Points redeemable for discounts at checkout
- Redemption rate and minimum threshold TBD by operator

The prototype shows a hardcoded challenge card (2 of 3 complete) and static points balance (450 pts). The NETLCO agent should build a proper `points_ledger` table in PostgreSQL with append-only event rows.
